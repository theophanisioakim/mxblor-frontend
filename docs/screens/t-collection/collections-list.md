# Screen SDD — Collections (the unit ledger)

> Screen design document — behaviour only, no implementation/API/data detail.
> Scoped to a single building; reached from the **Collections** tile on the
> [Edit Building](../building/edit-building.md) hub. Related to
> [Create Collection](./create-collection.md) and [Edit Collection](./edit-collection.md).

---

## 1. Screen

- **Name:** Collections
- **Location:** Buildings → *(a building)* → Collections
- **Purpose:** Show the unit ledger — what each unit has been charged and what it has paid — and let
  the user collect money from a unit.
- **Scope:** One flat grid of ledger rows, filtered by year, unit and collection type.

---

## 2. The model the screen shows — read this first

This is a **balance sheet**, mirroring the [supplier ledger](../t-payment/payments-list.md). Every row
is one side of it, and nothing is ever rewritten:

- Creating a distributed expense raises one **charge** row per unit — a **debit**, with a **due
  date** — for that unit's share, taken from the distribution table. The share is computed **once**,
  when the expense is created, and is never recomputed afterwards.
- Collecting inserts a **contra entry** — a **new credit** row, with **no due date**, carrying the
  **payment channel** and issuing a **receipt**. The charge row it was collected against is *not*
  touched.
- A unit's balance is therefore **derived**: `total debit − total credit`, across every row in that
  collection-type ledger — including its opening balance.

Two consequences the UI must not fight:

- **A unit may overpay.** Its balance goes negative, meaning the unit is in credit. That is a real
  state a managing agent needs to record, not an error to block.
- **A charge may be collected in instalments.** Each is its own credit row; there is no remainder row
  and no "partially paid" status.

### Opening balances (Balance B/F)

Every unit gets **two** zero opening-balance rows when it is created — one per collection-type ledger
(Capital, Monthly). They are what makes a unit's balance meaningful from day one. They are also the
only ledger rows a user sets directly: a **positive** amount means the unit owes, a **negative** one
means it is in credit. They can be edited only from the building-wide
[Building Unit Balances](../building-unit/building-unit-balances.md) screen; generic collection Add,
Edit, Delete, and bulk actions cannot change a Balance B/F row.

## 3. Entry points & navigation

- **Reached from:** the **Collections** tile on the [Edit Building](../building/edit-building.md) hub.
- **Breadcrumb:** Home › Buildings › *(building name)* › Collections.
- **Back:** returns to the parent building.

---

## 4. Grid columns

One row per ledger entry. Default sort: reference month, most recent first.

| Column | Content |
|---|---|
| Period | The reference month the row belongs to. |
| Unit | The building unit the row belongs to. |
| Expense | The catalog expense. Empty on an opening balance or a standalone row. |
| Collection type | Which ledger the row sits in (Capital / Monthly / …). |
| Debit | What the unit was charged. Zero on a collection. |
| Credit | What the unit paid. Zero on a charge. |
| Receipt no | The receipt issued when the money came in. Empty on a charge. |
| Payment channel | How the money came in — **set only on a collection**, which is how money-in rows are told apart. |

Paging 10 / 25 / 50 per page.

## 6. Actions

| Action | Placement | Availability | Outcome |
|---|---|---|---|
| **Collect** | Row action | Charge rows only (those with a due date) | Opens [Create Collection](./create-collection.md) to record money received. Inserts a contra entry, posts the bank movement, and **issues a receipt**. |
| **Add** | Above grid | All roles | Creates a **standalone** row not raised by any expense. |
| **Edit / Delete** | Row action | **Standalone rows only** | A row generated by an expense is edited or removed by editing or deleting **that expense**, never on its own. A contra entry that moved money cannot be edited — delete it (which voids its receipt) and re-collect. |
| **Back** | Above grid | All roles | Returns to the parent building. |

---

## 7. Receipts

A receipt is **issued, never authored**. Collecting mints the next number in that building's own
series — `<building code>-<six digits>`, e.g. `MXB-000042` — and snapshots the building code, building
name, unit code and payment channel onto it, so the document keeps saying what it said even if the
building is later renamed. Receipts cannot be created, edited or renumbered by hand; deleting the
collection deletes its receipt.

## 8. States

Loading / Loaded / Empty / failure — as standard.

## 9. Out of scope

- Per-year and per-month summary views.
- **Batch collection** (collecting a lump sum from one unit across several charges).
- Printing a receipt as a PDF (the reports module does not exist yet).
