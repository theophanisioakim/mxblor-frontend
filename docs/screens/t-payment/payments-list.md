# Screen SDD — Payments (the supplier ledger)

> Screen design document — behaviour only, no implementation/API/data detail.
> Scoped to a single building; reached from the **Payments** tile on the
> [Edit Building](../building/edit-building.md) hub. Related to
> [Create Payment](./create-payment.md) and [Edit Payment](./edit-payment.md).

---

## 1. Screen

- **Name:** Payments
- **Location:** Buildings → *(a building)* → Payments
- **Purpose:** Show the building's supplier ledger — what it owes and what it has paid — as a flat
  list of rows, and let the user pay against an outstanding obligation.
- **Scope:** One flat grid of ledger rows, filtered by year and expense.

---

## 2. The model the screen shows — read this first

This is a **balance sheet**, not a list of invoices with a status. Every row is one side of the
ledger, and nothing is ever rewritten:

- Creating a building expense raises an **obligation** row — a **credit**, with a **due date**. It
  says "we owe the supplier this".
- Paying inserts a **contra entry** — a **new debit** row, with **no due date**. It says "we paid
  this out". The obligation row it was paid against is *not* touched.
- What is still owed is therefore **derived**: `total credit − total debit`. There is no stored
  "outstanding" or "paid" flag, and there is no partial-payment bookkeeping — three instalments are
  simply three debit rows.

Two consequences the UI must not fight:

- **An obligation can be paid any number of times.** Each payment is its own row. There is no
  "already paid" state that closes it.
- **Overpaying is allowed.** The balance goes negative, which means the supplier owes *us*. That is a
  real state a managing agent needs to record, not an error to block.

---

## 3. Entry points & navigation

- **Reached from:** the **Payments** tile on the [Edit Building](../building/edit-building.md) hub.
- **Breadcrumb:** Home › Buildings › *(building name)* › Payments.
- **Back:** returns to the parent building.

---

## 4. Grid columns

One row per ledger entry. Default sort: reference month, most recent first.

| Column | Content |
|---|---|
| Period | The reference month the row belongs to. |
| Expense | The catalog expense. Empty on a standalone administrative row. |
| Credit | What we owe (an obligation raised by an expense). Zero on a payment. |
| Debit | What we paid out (a contra entry). Zero on an obligation. |
| Due date | Set on an obligation; **empty on a contra entry** — this is how the two are told apart. |
| Invoice no | Supplier invoice reference, if recorded. |
| Description | Auto-generated on a generated row, free text on a standalone one. |

Paging 10 / 25 / 50 per page.

---

## 6. Actions

| Action | Placement | Availability | Outcome |
|---|---|---|---|
| **Pay** | Row action | Obligation rows only (those with a due date) | Opens [Create Payment](./create-payment.md) to record money paid out. Inserts a contra entry and posts the bank movement. |
| **Add** | Above grid | All roles | Creates a **standalone** payment row not raised by any expense (an administrative payment). |
| **Edit / Delete** | Row action | **Standalone rows only** | A row generated by an expense carries a link back to it; it is edited or removed by editing or deleting **that expense**, never on its own. A contra entry that moved money cannot be edited — delete it and re-post. |
| **Back** | Above grid | All roles | Returns to the parent building. |

---

## 7. States

Loading / Loaded / Empty / failure — as standard.

---

## 8. Out of scope

- A per-year summary view (totals per year).
- Payments against **revenues** — the revenue ledger does not exist yet.
- Bulk payment (paying several obligations at once).
